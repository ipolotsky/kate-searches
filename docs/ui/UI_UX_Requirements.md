# UI/UX — Требования (внутренний стандарт)

> Нормативный документ команды. Применим к **любому** web-продукту/экрану (desktop и mobile web), без привязки к конкретному проекту. Здесь — только **измеримые требования** к интерфейсу: что должно быть, а что запрещено.

## 1. Назначение и область действия

- **Документ описывает** продуктовые требования к пользовательскому интерфейсу: доступность, эвристики, мобильный UX, онбординг, микрокопи, состояния и консистентность.
- **Документ не описывает** процесс оценки (эвристический аудит, SUS, think-aloud, ИИ-агенты, HEART-метрики) — это отдельный регламент оценки.
- **Применяется** ко всем новым экранам, редизайнам и фичам до прохождения дизайна/ревью и релиза.
- **Приоритет при конфликте:** WCAG 2.2 (уровень AA — обязательный минимум, AAA — рекомендуется где возможно) > эвристики Нильсена > мобильные best practices > микрокопи > визуальная эстетика.
- **Юрисдикция документа** шире одного репо: распространяется на web-продукты команды (Next.js/Flowbite-React или иные стеки).

## 2. Глоссарий

| Термин                              | Определение                                                                                                                     |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| **Результативность (ISO 9241-11)**  | Способность пользователя достичь цели корректно и полностью.                                                                    |
| **Эффективность (ISO 9241-11)**     | Соотношение затраченных усилий (время, когнитивная нагрузка) к достигнутому результату.                                         |
| **Удовлетворённость (ISO 9241-11)** | Субъективное позитивное восприятие процесса, включая доверие и эстетический комфорт.                                            |
| **Юзабилити**                       | Измеримая степень, в которой продукт usable конкретными пользователями в конкретном контексте. Не «удобство», а 3 метрики выше. |
| **Touch target**                    | Кликабельная область интерактивного элемента (кнопка, ссылка, иконка-экшен).                                                    |
| **Thumb zone**                      | Нижняя треть экрана смартфона, доступная большим пальцем без перехвата устройства.                                              |
| **Aha-момент**                      | Точка, в которой пользователь впервые осознаёт ценность продукта через действие.                                                |
| **Progressive disclosure**          | Раскрытие сложности по мере необходимости: не вываливать все опции сразу.                                                       |
| **Микрокопи**                       | UI-тексты: кнопки, лейблы, подсказки, ошибки, тултипы, пустые состояния.                                                        |
| **Skeleton screen**                 | Экран-скелет с плейсхолдерами контента вместо спиннера для состояния загрузки.                                                  |
| **Happy path**                      | Идеальный сценарий без ошибок/возвратов/отвлечений — НЕ исчерпывает реальное поведение.                                         |
| **Тенант-изоляция**                 | Неприменимо к этому документу (здесь только UI).                                                                                |

## 3. Доступность (WCAG 2.2 — обязательный минимум AA)

Все критерии с уровнем `AA` обязательны. `AAA` рекомендуется где достижимо без ущерба для задач.

### 3.1 Таблица обязательных критериев WCAG 2.2

| ID          | Критерий WCAG 2.2                 | Уровень | Требование                                                                                                                                                     | AC        |
| ----------- | --------------------------------- | ------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------- |
| **REQ-A01** | Focus Not Obscured (2.4.11)       | AA      | При навигации с клавиатуры индикатор фокуса **не перекрывается** sticky-хедером, оверлеем, ф standoff-элементами.                                              | AC-UX-A01 |
| **REQ-A02** | Focus Appearance (2.4.13)         | AA      | Индикатор фокуса визуально различим, толщина контура ≥ **2px**, контраст к фону ≥ 3:1. Никогда не убирается через `outline: none` без замены.                  | AC-UX-A02 |
| **REQ-A03** | Dragging Movements (2.5.7)        | AA      | Любая функция с drag-and-drop / сложными жестами имеет альтернативу кликом/кнопками со стрелками. Drag-only взаимодействия **запрещены**.                      | AC-UX-A03 |
| **REQ-A04** | Target Size Minimum (2.5.8)       | AA      | Минимум 24×24 CSS px. Best practice — **44×44 px** (iOS) / **48 dp** (Android) для всех touch targets.                                                         | AC-UX-A04 |
| **REQ-A05** | Consistent Help (3.2.6)           | A       | Справка/контакты/чат поддержки расположены **единообразно** на всех экранах (один паттерн и место).                                                            | AC-UX-A05 |
| **REQ-A06** | Accessible Authentication (3.3.8) | A       | Запрещён вход, опирающийся исключительно на когнитивный тест (запоминание пароля, визуальная CAPTCHA) без альтернативы (биометрия, email-link, WebAuthn, SSO). | AC-UX-A06 |
| **REQ-A07** | Контраст текста                   | AA      | ≥ 4.5:1 для основного текста, ≥ 3:1 для крупного (≥18pt / ≥14pt bold). UI-компоненты и графика — ≥ 3:1.                                                        | AC-UX-A07 |
| **REQ-A08** | Работа с клавиатуры               | A       | Любая функция доступна через Tab/Enter/Space/Arrows без мыши. Запрет `hover-only` действий (tooltip-only-trigger, hover-only-menus).                           | AC-UX-A08 |
| **REQ-A09** | Skip-link и лендинг Tab-focus     | A       | Первые Tab-фокусы — skip-link «к основному контенту»; видимый фокус на первой интерактивной точке.                                                             | AC-UX-A09 |
| **REQ-A10** | Не полагаться на цвет             | A       | Состояние (ошибка/успех/обязательное) кодируется не только цветом: иконка, текст, толщина, граница.                                                            | AC-UX-A10 |
| **REQ-A11** | ARIA-роль/имя/значение            | A       | Все интерактивные элементы имеют корректную role + доступное имя (`aria-label`/`aria-labelledby`/видимый текст) + состояние через `aria-`\*.                   | AC-UX-A11 |
| **REQ-A12** | Семантика форм                    | A       | Все `<input>` имеют `<label>` (или `aria-label`); association `for`/`id`; автозаполнение через `autocomplete`.                                                 | AC-UX-A12 |
| **REQ-A13** | Уважение prefers-reduced-motion   | AA      | Несущественные анимации отключаются при `prefers-reduced-motion: reduce`.                                                                                      | AC-UX-A13 |
| **REQ-A14** | Reflow и адаптивность             | AA      | При 320 CSS px ширины нет горизонтального скролла; контент reflows в одну колонку.                                                                             | AC-UX-A14 |
| **REQ-A15** | Пауза/стоп анимаций               | A       | Любая анимация длительностью > 5s или мигание — наличие паузы/стопа.                                                                                           | AC-UX-A15 |

### 3.2 Скринридер-минимум

- Последовательность DOM = последовательность визуального чтения (не нарушать порядок фокуса через `tabindex`- abuse).
- Каждая страница имеет `<h1>` (один), иерархия заголовков без скачков (h1 → h2 → h3).
- Динамические обновления (toast, inline-ошибки, изменения статуса) анонсируются через `aria-live` (`polite` по умолчанию, `assertive` для критичных ошибок).
- Изображения: decorative → `alt=""`, meaningful → короткое `alt`, сложные → `aria-describedby` к подписи.

## 4. Эвристики Нильсена как требования (10 принципов)

Каждый принцип переформулирован как observable требование с ID.

### H1. Видимость статуса системы

**REQ-H01.** Любая асинхронная операция > 300 мс показывает индикатор состояния (spinner/skeleton/progress bar/inline-status). Запрещены «висящие» состояния без признаков процесса.
**REQ-H02.** Переходы по иерархии (категория→детали) сопровождают breadcrumb или явной навигационной метой «где я сейчас».
**REQ-H03.** Сохранение/Отправка/Удаление показывают явный success-сигнал (toast, badge, inline-confirmation), не ожидают от пользователя догадаться.

### H2. Совпадение системы с реальным миром

**REQ-H04.** Язык интерфейса — язык аудитории; запрещены жаргон, системные коды (`Error 404`, `NullPointerException`), технические сообщения от internal logging. Каждое сообщение о ошибке — человекочитаемо.
**REQ-H05.** Знакомые метафоры: иконка лупы = поиск; шестерёнка = настройки; корзина = удаление; домик = главная. Запрет на неочевидные/авторские иконки без подписи.

### H3. Контроль и свобода пользователя

**REQ-H06.** Каждая многошаговая операция (онбординг, визард, чекаут) имеет **видимый** «Назад». Запрет сцен без отмены.
**REQ-H07.** Каждое деструктивное действие (удалить, отменить подписку, сбросить) требует **confirm-dialog** с явной формулировкой последствий и кнопкой отмены.
**REQ-H08.** Модальные окна закрываются Esc, кликом по backdrop, кнопкой × в углу; фокус возвращается на триггер.
**REQ-H09.** Richtext/формы с большой правкой имеют предупреждение при попытке покинуть страницу с несохранёнными данными.

### H4. Единообразие и стандарты

**REQ-H10.** Один UI-паттерн для одной функции: один стиль primary-кнопки, один стиль input, один tooltip-паттерн, одна навигация.
**REQ-H11.** Запрет синонимичных иконок/слов для одного действия (нельзя: «Сохранить» и «Применить» в одном контексте).
**REQ-H12.** Деструктивные действия — красный/оранжевый предупреждающий цвет; положительные — зелёный/синий. Запрет инверсии.

### H5. Предотвращение ошибок

**REQ-H13.** Все input с жёстким форматом (дата, телефон, email, card) имеют валидацию формата + автозаполнение (`autocomplete`) + правильный тип клавиатуры (`type=email`, `type=tel`, `inputmode`).
**REQ-H14.** Необратимые операции (удаления, публикация, отправкаMass) — двух-шаговое confirm с показом затронутых сущностей.
**REQ-H15.** Запрет кнопок «Отправить» в активном состоянии на невалидной форме; disable при невалидности либо inline-сообщение.
**REQ-H16.** Field-level валидация в реальном времени; запрет валидировать форму целиком только на submit (только если это 안тивigned out отдельно).

### H6. Распознавание, а не вспоминание

**REQ-H17.** Опции подсказываются, а не вкладываются в память пользователя: dropdowns/combobox с поиском, недавно просмотренные элементы, history, suggestions.
**REQ-H18.** Сессия показывает «где я»: выбранный пункт меню подсвечен, breadcrumb актуален, тайтл страницы соответствует контенту.
**REQ-H19.** Передача данных между шагами визарда — видима (review-страница перед submit) либо явно оповещается.

### H7. Гибкость и эффективность использования

**REQ-H20.** Чаще используемые действия доступны в ≤ 2 кликах; глубже 3 кликов — требуется обоснование.
**REQ-H21.** Горячие клавиши для опытных (Cmd/Ctrl+Enter = отправить, Esc = закрыть модалку,/? = help) document-ируются в подсказке.
**REQ-H22.** Длинные списки поддерживают фильтр/поиск/сортировку; пагинация/бесконечный скролл с явной меткой о загрузке.

### H8. Эстетика и минимализм

**REQ-H23.** Каждая видимая единица контента релевантна задаче экрана; нефильтр-неопр-не-need-info выносится в подуровень (details/accordion/secondary-таб).
**REQ-H24.** MVP-визард: один primary CTA на экран. Вторичные действия — secondary-стиль, не конкурируют по весу.

### H9. Помощь в диагностике и исправлении ошибок

**REQ-H25.** Сообщение об ошибке строится по формуле **«Что произошло + Почему + Что делать»**: `«Не удалось сохранить. Сервер недоступен. Нажмите «Повторить» через минуту.»`. Запрет `Error 500` / `Something went wrong` без конкретики.
**REQ-H26.** Ошибка валидации поля — inline рядом с полем + атрибут `aria-invalid` + привязка через `aria-describedby`. Запрет только toast для field-ошибок.
**REQ-H27.** После неудачи доступен явный «Повторить» на ошибке, не требует от пользователя полного перезаполнения формы (сохранять введённые данные).

### H10. Доступная справка и документация

**REQ-H28.** Справка/FAQ/docs доступны с любой страницы в одном и том же месте (WCAG 3.2.6, REQ-A05) и не глубже 2 кликов.
**REQ-H29.** Контекстные тулзы (tooltips на сложных полях, `?` рядом с labelled-термином) — рядом с местом, где они нужны; не в отдельном разделе help.

## 5. Мобильный web-UX

| ID          | Требование                                                                                                                                                                                           | AC        |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------- |
| **REQ-M01** | **Thumb zone**: все primary CTА и primary-навигация располагаются в нижней трети экрана. Запрет primary CTA в правом верхнем углу.                                                                   | AC-UX-M01 |
| **REQ-M02** | Нижний bar навигации — **не более 5 пунктов**; каждая иконка имеет текстовую подпись (запрет иконок-only в primary-nav).                                                                             | AC-UX-M02 |
| **REQ-M03** | Touch targets ≥ 44×44 px (iOS) / 48 dp (Android). WCAG-минимум 24×24 — не ниже; при сложных рядах — spacing ≥ 8px между соседними кликабельными.                                                     | AC-UX-M03 |
| **REQ-M04** | First contentful paint / готовы к взаимодействию ≤ 3 с на 4G (для ключевых экранов). Используются skeleton screens для списков > 200 мс загрузки.                                                    | AC-UX-M04 |
| **REQ-M05** | Прогрессивная загрузка: тяжёлые блоки / изображения загружаются lazy; контент работоспособен при медленном/ограниченном соединении (отображаются данные из кэша / offline-first для критичных read). | AC-UX-M05 |
| **REQ-M06** | Формы: `type`/`inputmode` → корректная клавиатура (tel/email/url/numeric); `autocomplete` согласно WHATWG; `enterkeyhint` для кнопки Enter на мобильном.                                             | AC-UX-M06 |
| **REQ-M07** | Inline-валидация в реальном времени (на blur, не на каждый keystroke); результаты видны над клавиатурой.                                                                                             | AC-UX-M07 |
| **REQ-M08** | Sticky-элементы (хедеры/футеры/CTA) не перекрывают контент и фокус в верхней/нижней видимой области (REQ-A01).                                                                                       | AC-UX-M08 |
| **REQ-M09** | Не использовать hover-only взаимодействия (touch не имеет hover); tap-and-hold必须有 кнопочная alternative (REQ-A03).                                                                                | AC-UX-M09 |
| **REQ-M10** | Разрывы сессии минимальны: при прерывании (входящий звонок, переключение приложений) введённые данные сохраняются.                                                                                   | AC-UX-M10 |

## 6. Онбординг (First-Time User Experience)

| ID          | Требование                                                                                                                                       | AC        |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------ | --------- |
| **REQ-O01** | Длина онбординга — **≤ 3 экранов** (визард/слайдов). Длинные обучающие карусели запрещены.                                                       | AC-UX-O01 |
| **REQ-O02** | **Progressive disclosure**: сложные функции открываются после базового «Aha-момента», а не в один заход. Один концепт — один экран.              | AC-UX-O02 |
| **REQ-O03** | Ценность (interactive-действие; демонстрация результата) показывается **до** регистрации и **до** запросов разрешений (геолокация, пуш, камера). | AC-UX-O03 |
| **REQ-O04** | Регистрация откладывается до момента, когда без неё нельзя получить первую реальную ценность. Запрет registration-only-entry.                    | AC-UX-O04 |
| **REQ-O05** | Кнопка **«Пропустить»** (Skip) — заметная, доступна в каждой точке онбординга; ведёт сразу к функционалу.                                        | AC-UX-O05 |
| **REQ-O06** | Разрешения (push/geo/camera) запрашиваются по-одному (по 1 за экран), в контексте полезности; не при старте приложения «пачкой».                 | AC-UX-O06 |
| **REQ-O07** | Прогресс-индикатор шагов визарда — видим (точки/бар «2 из 3»); пользователь знает, сколько осталось.                                             | AC-UX-O07 |
| **REQ-O08** | На каждом шаге — primary-кнопка «Далее» в Thumb Zone (REQ-M01), secondary «Назад» — ниже по иерархии.                                            | AC-UX-O08 |
| **REQ-O09** | Не требовать обязательного ввода сразу нескольких типов данных (контакты, профиль, preferencies) на одном экране — дробить по consent.           | AC-UX-O09 |
| **REQ-O10** | После Skip онбординг доступен «дойти позже» в настройках/help; не закрыт окончательно.                                                           | AC-UX-O10 |

## 7. Микрокопи

| ID          | Требование                                                                                                                                                                         | AC        |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------- |
| **REQ-C01** | Запрет технического жаргона и кодов в пользовательских текстах. Человекочитаемая переформулировка обязательна.                                                                     | AC-UX-C01 |
| **REQ-C02** | **Формула сообщения об ошибке:** `Что произошло + почему + что делать`. Запрет тупиковых формулировок (`Ошибка`, `Something went wrong`, `Invalid input`).                         | AC-UX-C02 |
| **REQ-C03** | **Позитивный тон** микрокопи: ошибка — часть пути, не «виноват пользователь». Замена `«Неверный ввод»` → `«Похоже, вкралась опечатка. Давайте проверим срок действия карты.»`.     | AC-UX-C03 |
| **REQ-C04** | **Превентивные приписки** под чувствительными полями: под телефоном — `«Используем только для статуса доставки»`; под email — `«Спам не пришлём, отпишетесь в один клик»`.         | AC-UX-C04 |
| **REQ-C05** | CTA на кнопках отражают **действие и результат**, не абстракцию: `«Сохранить и продолжить»` вместо `«ОК»`; `«Удалить черновик»` вместо `«Подтвердить»`.                            | AC-UX-C05 |
| **REQ-C06** | Для деструктивных действий — конкретные последствия в тексте кнопки / confirm: `«Удалить 7 черновиков?»` вместо `«Удалить выбранные»`.                                             | AC-UX-C06 |
| **REQ-C07** | Микрокопи локализуется (i18n) — не хардкод в JSX, всё через translations-bundle. Запрет склейки строк с подстановкой: `«Удалено {n} {элементов»}` — использует plural-rules (ICU). | AC-UX-C07 |
| **REQ-C08** | Плейсхолдер input ≠ замена label (`placeholder` исчезает после ввода); label обязателен. Плейсхолдер — короткий пример формата (`ivan@example.com`).                               | AC-UX-C08 |
| **REQ-C09** | Empty states — не «Здесь пусто», а **якорь на действию**: `«У вас пока нет источников. Добавьте первый →»` + CTA.                                                                  | AC-UX-C09 |
| **REQ-C10** | Loading-тексты для > 5 с операций — сущность процесса (`«Собираем свежие источники…»`), не просто `«Загрузка…»`.                                                                   | AC-UX-C10 |

## 8. Состояния системы и обратная связь

**REQ-S01.** У каждого списка/таблицы/ленты реализованы состояния: **loading / empty / error / partial / success (with-items)**. Запрет «висящих» состояний.
**REQ-S02.** Loading > 300 мс → skeleton/индикатор; > 5 с → контекстный длинный-progress / объяснение.
**REQ-S03.** Empty state имеет: иконку/иллюстрацию, краткое объяснение, CTA для выхода из пустого состояния.
**REQ-S04.** Error state: что произошло (REQ-C02), кнопка «Повторить», ссылка на help/контакты (REQ-A05).
**REQ-S05.** Async form-submit: кнопка блокируется + меняет label (`«Сохранение…»`), результат (success/error) анонсируется через `aria-live`.
**REQ-S06.** Optimistic UI для быстрых операций (toggle/звезда): мгновенный ответ + rollback ниже при ошибке.
**REQ-S07.** Indeterminate-progress запрещён для операций > 10 с: должен быть счетчик времени/шагов либо заменён на determinate-progress.
**REQ-S08.** Toast`ы не перекрывают primary CTA и primary-навигацию; auto-dismiss ≥ 4 с для reading; action-toast (`«Отменить»`) сохраняется дольше;` role="status"`, не блокируют основной поток.

## 11. Acceptance criteria (общий формат)

Каждая таблица выше привязана к AC через колонку `ID` и помечает проверяемость. Формат:

- **AC-UX-<буквараздела><2-значный>**: «На экране X каждая [объект требования REQ-...] соответствует [полное формулирование observable-критерия], проверяется [способ]».
- Способ проверки (criteria verification method): **ручной дым** / **design-review** / **автотест (playwright/axe)** / **devtools audit (Lighthouse/Wave)** / **скринридер (VoiceOver/NVDA)**.
- Блокеры — нарушение REQ-Xxx (стоп-лист). High — REQ-Axx, REQ-Hxx. Medium — REQ-Mxx, REQ-Oxx, REQ-Cxx. Low — косметика (REQ-Kxx в части spacing).

### Пример AC

- **AC-UX-A03**: «Любая функция с drag-and-drop имеет альтернативу кликом/стрелками; проверяется ручным дымом экранов с drag-сценариями и axe/playwright-тестом доступности». **Блокер при невыполнении.**
- **AC-UX-M04**: «Первые критичные экраны достигают состояния готовности к взаимодействию ≤ 3 с на 4G; проверяется Lighthouse Performance Budget\*\*.
- **AC-UX-O01**: «Онбординг проекта содержит ≤ 3 экранов; проверяется design-review + число `wizard-step` компонентов ≤ 3 в коде». **Блокер при невыполнении.**
- **AC-UX-C02**: «Все пользовательские сообщения об ошибках содержат часть `что произошло + почему + что делать`; проверяется grep по коду (`/(Something went wrong |Error \d+ |Invalid input)/` без контекста = fail)». **High.**
