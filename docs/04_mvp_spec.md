# 04 — MVP-спецификация

Цель MVP: довести один живой конвейер до качества, при котором маркетолог LOOTON **реально публикует** черновики и экономит часы. Всё остальное — после подтверждения, что движки отбора и генерации дают приемлемое качество.

## 1. Принцип реза

> **MVP = вертикальный срез через весь конвейер для немногих источников, а не широкий охват с сырыми движками.**

Лучше 10 источников с точным скорингом и хорошими черновиками, чем 100 источников и мусор на выходе. Качество отбора и голоса бренда — это и есть продукт; всё вокруг (биллинг, соцсети, автопубликация) — обвязка.

## 2. В скоупе MVP / вне скоупа

### ✅ В скоупе
- Self-serve регистрация + онбординг (мультитенант, RLS с D1).
- Профиль бренда: описание компании/аудитории, критерии фильтра (свободный текст), voice-конфиг + примеры постов (few-shot), загрузка файлов.
- Управление источниками: добавление URL, тип `rss` / `scraper`, тест источника, приоритет/категория.
- Адаптеры: **RSS + news-sitemap + Crawl4AI-скрапер** (Firecrawl-fallback). Абстракция адаптера готова под соцсети.
- Daily-пайплайн: ingest → extract → novelty+dedup → score → generate.
- Скоринг по критериям тенанта с объяснением (rubric из доков Kate).
- Генерация черновика «новость × продукт» в голосе бренда + SEO/AEO-инструкции + JSON-LD.
- Дашборд черновиков со статусами (`Новый`/`В работе`/`Опубликован`/`Отклонён`/`Архив`), сортировка «сначала новые».
- Редактор поста (правка тела, заголовка, SEO-блока).
- Обратная связь по скорингу и по черновику (rating + правки → копим).
- **Cost-metering per tenant** (LiteLLM + Langfuse + зеркало в `ai_usage`), отображение расхода.
- Базовая админка: список тенантов, AI-расходы, ручное переключение плана/бюджета/порога.

### ❌ Вне скоупа MVP (роадмап)
- Stripe-биллинг и платный self-checkout (фаза 1.5 — сразу после MVP).
- Соцсеть-адаптеры (Telegram/Reddit — фаза 2; IG/TikTok/X — фаза 3 через OAuth).
- Автопубликация в CMS (MVP — экспорт markdown/HTML + copy). Интеграция WordPress/Webflow/Ghost — фаза 2.
- Автоматический re-train промптов из фидбэка (MVP — копим фидбэк + ручной few-shot инжект; авто-eval-цикл — фаза 2).
- Семантический дедуп на эмбеддингах (MVP — URL-канон + hash + simhash).
- Привязка к товарам через API маркетплейса (MVP — опционально по URL каталога/ручной список).
- Мульти-язык в полном объёме (MVP — RU+EN для LOOTON, но архитектурно язык — поле).
- AI Visibility-трекинг (попадание в AI-ответы) — фаза 3.

## 3. Эпики и user stories

### EPIC A — Онбординг и профиль
- A1. Как новый пользователь, я регистрируюсь (email/OAuth) и создаю тенант, чтобы начать.
- A2. Как owner, я заполняю профиль бренда (компания, аудитория, критерии, тон, примеры постов + откуда инфоповод), чтобы движки настроились под меня.
- A3. Как owner, я загружаю файлы (брендбук, примеры статей), чтобы обогатить контекст.
- A4. Как owner, я добавляю/редактирую источники (URL, тип, приоритет, категория) и **тестирую** источник, чтобы убедиться, что он парсится.

### EPIC B — Сбор и отбор
- B1. Как система, я раз в сутки прохожу источники тенанта и беру только новости за сегодня.
- B2. Как система, я извлекаю чистый текст, канонизирую URL, дедуплицирую.
- B3. Как система, я скорю каждую новость по критериям тенанта и сохраняю объяснение.
- B4. Как система, прошедшие порог — отправляю на генерацию; остальные помечаю `filtered_out` (но они доступны для просмотра/фидбэка).

### EPIC C — Генерация и дашборд
- C1. Как система, для прошедших отбор генерю черновик в голосе бренда + SEO/AEO + JSON-LD + привязку к товарам.
- C2. Как маркетолог, утром вижу дашборд: сначала новые черновики, карточка показывает источник (ссылка), скор+объяснение, какой AI, заголовки, товары.
- C3. Как маркетолог, открываю черновик в редакторе и правлю текст/заголовок/SEO-блок.
- C4. Как маркетолог, меняю статус: В работе / Опубликован / Отклонён / Архив.
- C5. Как маркетолог, экспортирую финал (markdown/HTML + JSON-LD) для публикации.
- C6. Как маркетолог, вижу подробные SEO/семантические инструкции к посту.

### EPIC D — Обратная связь и самообучение
- D1. Как маркетолог, ставлю оценку и комментарий по релевантности новости (скоринг попал/мимо).
- D2. Как маркетолог, мои правки черновика сохраняются как дельта (сигнал качества).
- D3. Как система, коплю фидбэк; одобренные/правленные посты идут в per-tenant few-shot store (ручной инжект в MVP).

### EPIC E — Метеринг и админка
- E1. Как система, мерю $ каждого AI-запроса в разрезе тенанта/стадии.
- E2. Как owner, вижу свой расход и сколько % бюджета потрачено.
- E3. Как admin, вижу всех тенантов, их метрики и AI-расходы.
- E4. Как admin, вручную меняю план/бюджет/порог апселла тенанта.
- E5. Как система, при расходе ≥ N% бюджета показываю предложение апгрейда.

## 4. Экраны (UI на Flowbite)

| Экран | Ключевые элементы |
|---|---|
| **Auth** | Регистрация/логин (Supabase Auth), создание тенанта |
| **Онбординг (визард)** | Шаги: компания → аудитория → критерии → голос+примеры → источники. Прогресс-бар |
| **Дашборд черновиков** | Колонки/список со статусами; сортировка «новые → в работе → архив»; карточка: источник-ссылка, скор-бейдж, модель, заголовок, превью, кнопки edit/publish/kill |
| **Редактор поста** | Markdown-редактор тела; панель SEO/AEO (meta, headings, FAQ, ключи, JSON-LD предпросмотр); привязанные товары; кнопка экспорта; блок фидбэка |
| **Исходник новости** | Оригинал (ссылка/текст) + объяснение скоринга по критериям; кнопки фидбэка по релевантности |
| **Профиль/настройки** | Бренд-профиль, голос+примеры, файлы, список источников + тест |
| **Биллинг/использование** | Текущий расход, % бюджета, плашка апселла (без оплаты в MVP) |
| **Админка** | Таблица тенантов (план, MAU, drafts, AI $), детальная карточка тенанта, переключатели плана/бюджета/порога, отчёт по расходам |

## 5. Статусы поста (машина состояний)

```
new ──(взять в работу)──> in_progress ──(опубликовать)──> published
 │                              │
 ├──(отклонить)──> rejected     ├──(в архив)──> archived
 └──(в архив)──> archived       └──(вернуть)──> new
```

Дашборд по умолчанию: `new` сверху (фокус), потом `in_progress`, `archived`/`published`/`rejected` — в свёрнутых секциях/фильтрах.

## 6. Acceptance criteria (ключевые, измеримые)

- **AC-1 (новизна):** пайплайн не создаёт черновики из статей с `published_at` старше «сегодня» (по таймзоне тенанта). Дубли (один canonical_url) не создаются дважды.
- **AC-2 (скоринг):** каждая новая статья получает `relevance_score` (0–100) + структурированное объяснение по критериям; решение pass/fail по настраиваемому порогу.
- **AC-3 (генерация):** для прошедших отбор создаётся `post` с телом, ≥3 предложенными заголовками, FAQ, валидным JSON-LD (`Article` + `FAQPage`), SEO-инструкциями.
- **AC-4 (метеринг):** для каждого LLM-вызова в `ai_usage` записан tenant_id, stage, model, токены, cost_usd; сумма по тенанту совпадает с Langfuse (±1%).
- **AC-5 (изоляция):** пользователь тенанта A не может прочитать/изменить данные тенанта B (проверяется тестом RLS).
- **AC-6 (дашборд):** сортировка и фильтр по статусу работают; «новые» всегда сверху.
- **AC-7 (фидбэк):** оценка и правки сохраняются и привязаны к article/post.
- **AC-8 (апселл):** при `ai_spent ≥ threshold%` показывается плашка апгрейда.

## 7. Вехи (ориентир для 1–2 инженеров)

> Сроки — грубая оценка, зависят от команды. Логика — порядок, а не календарь.

| Веха | Содержание | Грубо |
|---|---|---|
| **M0 — Каркас** | Репо (Next.js + FastAPI), Supabase (схема + RLS), auth, деплой-пайплайн, LiteLLM+Langfuse подняты | 1–1.5 нед |
| **M1 — Ingestion** | RSS+sitemap+Crawl4AI адаптеры, extract+dedup+novelty, таблица articles, тест источника в UI | 1.5–2 нед |
| **M2 — Скоринг** | Промпт-движок скоринга (rubric из доков Kate), Instructor-схема, объяснение, порог, метеринг | 1–1.5 нед |
| **M3 — Генерация** | Промпт-движок драфтинга (голос+SEO/AEO+JSON-LD), привязка товаров (опц.), посты | 1.5–2 нед |
| **M4 — UI** | Дашборд, редактор, профиль/онбординг, исходник+объяснение, фидбэк | 2–2.5 нед |
| **M5 — Админка + метеринг-репорты** | Список тенантов, AI-расходы, переключатели, апселл-плашка | 1–1.5 нед |
| **M6 — Пилот LOOTON** | Залить её источники+критерии+примеры, прогнать неделю, откалибровать пороги/промпты по фидбэку | 1–2 нед |

**Definition of Done MVP:** LOOTON получает каждый рабочий день очередь релевантных черновиков, публикует ≥1/день с правкой <30%, мы видим AI-расход на неё в админке, фидбэк копится.
